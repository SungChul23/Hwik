### 💡 기획 의도

현대인의 휴식 시간, 버스와 지하철에서의 자투리 시간은 '유튜브 쇼츠'나 '릴스'가 점령했습니다. 우리는 **"사람들이 멍하니 흘려보내는 이 시간을 학습 시간으로 바꿀 수는 없을까?"** 라는 질문에서 출발했습니다.

* **부담 없는 접근:** 공부라는 심리적 장벽을 없애기 위해, 사용자에게 가장 익숙한 **숏폼 UI**를 차용했습니다.

* **강력한 훅(Hook):** "교육용 영상은 지루하다"는 편견을 깨기 위해, 에듀테크 관점에서 **몰입감 높고 흥미로운 시나리오의 숏폼 콘텐츠**를 직접 제작하여 학습 지속성을 확보했습니다.

### 2. 기술 전략

#### Full Serverless Backend (AWS Lambda + API Gateway)
* **서버리스 전환:** 고정 비용이 발생하는 EC2 대신, **AWS Lambda**와 **API Gateway**를 활용하여 모든 비즈니스 로직 처리.
* **API 구성:** 총 **28개의 API Endpoints**를 마이크로서비스 단위로 분리 구현, 특정 기능 트래픽 급증 시에도 전체 서비스 안정성 확보.

#### Gen-AI Pipeline Automation (Step Functions Workflow)
**"매력적인 숏폼 콘텐츠"**의 지속적 공급을 위해 **AWS Step Functions** 기반의 영상 및 학습 데이터 생성 파이프라인 자동화 구현.

* **영상 생성 파이프라인**
    * **리소스 확보:** `Google Custom Search` 기반 이미지 크롤링 및 **Google Veo 3** 모델 활용을 통한 고품질 영상 생성 유도.
    * **품질 검증 (QA):** 각 단계별 **Lambda 기반 검수 로직** 배치, 교육용 부적합 판정 시 자동 재생성 루프(Loop) 수행.
* **학습 데이터 생성 파이프라인**
    * **가치 보장:** **엄격한 가이드라인**이 적용된 프롬프트를 통해 학습 질문의 교육적 가치 및 정확성 확보.

#### Advanced AI Integration
* **AI 피드백 (Bedrock):** **AWS Bedrock (Claude 모델)** 연동을 통한 사용자 발화 분석 및 자연스러운 피드백 제공.
* **음성 처리 (Polly & Transcribe)**
    * **AWS Polly:** 원어민 수준의 자연스러운 억양을 가진 TTS 제공으로 리스닝 품질 제고.
    * **AWS Transcribe:** 사용자 음성의 텍스트 변환(STT) 및 발음 정확도 분석을 위한 기초 데이터 생성.

#### Global Latency Optimization (CloudFront)
* **문제 상황:** 미국 동부(N. Virginia) 리전 고정으로 인한 한국 사용자의 영상 로딩 지연(Latency) 발생 우려.
* **해결책:** **AWS CloudFront (CDN)** 도입, 정적 리소스(영상, 이미지)의 엣지 로케이션 캐싱 처리를 통해 물리적 한계 극복 및 **끊김 없는 숏폼 스트리밍 환경 제공**.

# 🤔 문제해결

## 트러블 슈팅 

프로젝트 개발 과정에서 직면한 기술적 난제와, 이를 엔지니어링 관점에서 해결한 경험입니다.

<details>
<summary><b> 1. 데이터 흐름 최적화 및 동적 렌더링 아키텍처 (Data Flow & Scalability)</b></summary>
<div markdown="1">
<br>

**1) 문제 상황: API 중복 호출과 확장성의 한계**
* **성능 이슈 (Performance):** 초기 설계에서는 페이지 이동 시마다 API를 새로 호출했습니다. 이로 인해 화면 전환 시 불필요한 로딩(Latency)이 발생하고, 모바일 웹 환경에서 데이터 소모량과 서버 비용이 증가하는 오버헤드(Overhead) 문제가 있었습니다.
* **유지보수 이슈 (Maintainability):** 12개 이상의 숏폼 콘텐츠를 제공하기 위해 각각의 페이지를 정적으로 생성했습니다. 콘텐츠가 늘어날 때마다 페이지를 추가해야 하는 비효율적인 구조로, 서비스 확장 시 유지보수 비용이 선형적으로 증가하는 한계에 봉착했습니다.

**2) 해결 전략: Client-side Data Passing & Dynamic View Template**
* **State Management 최적화:** API 응답으로 받은 대용량 JSON 데이터를 파싱한 후, `Page Parameter`를 활용해 다음 페이지로 객체(Object) 자체를 넘겨주는 **Data Passing 패턴**을 정립했습니다. 이를 통해 화면 전환 시 중복 API 호출을 **100% 제거**하고, 즉각적인(Zero-latency) 페이지 전환을 구현했습니다.
* **Single Template Architecture:** N개의 콘텐츠를 하나의 동적 페이지(Dynamic Page)에서 처리할 수 있도록 템플릿 구조로 개편했습니다. 데이터의 유무(Null Safety)에 따라 UI가 유동적으로 변하는 **조건부 렌더링(Conditional Visibility)** 로직을 적용하여, 단 하나의 페이지로 무한한 콘텐츠 확장이 가능한 유연한 아키텍처를 완성했습니다.

**3) 결과**
* 콘텐츠가 수백 개로 늘어나도 코드 수정 없이 즉시 대응 가능한 **확장성(Scalability)** 확보.
* 네트워크 의존도를 낮춰 사용자 경험(UX)을 네이티브 앱 수준으로 향상.

</div>
</details>

<details>
<summary><b> 2. 웹/앱 환경 차이 극복 및 미디어 처리 최적화 (Cross-Platform & Byte Manipulation)</b></summary>
<div markdown="1">
<br>

**1) 문제 상황: 브라우저 엔진 파편화(Fragmentation)와 호환성 이슈**
* **코덱 호환성(Codec War):** 웹 배포 환경에서 Android(Chrome 엔진)는 고효율의 `WEBM_OPUS` 포맷을 지원하지만, iOS(WebKit 엔진)는 이를 지원하지 않고 비압축 방식인 `LINEAR16 (WAV)` 포맷만을 강제하여 오디오 재생 및 녹음에 치명적인 오류가 발생했습니다.
* **데이터 규격 불일치:** iOS 웹 환경은 하드웨어적으로 **Stereo(2ch)** 녹음이 기본이지만, 서버 측 STT(Speech-to-Text) AI 모델은 정확도를 위해 **Mono(1ch)** 입력을 요구하여 인식률이 현저히 떨어지는 문제가 있었습니다.
* **Autoplay 정책 충돌:** iOS Safari의 엄격한 'Autoplay Policy'로 인해, 숏폼의 핵심인 '소리 포함 자동 재생'이 차단되어 사용자 몰입을 방해했습니다.

**2) 해결 전략: Client-side Pre-processing & User-Agent Handling**
* **Binary Level Processing (전처리):** 서버의 부담을 줄이고 호환성을 확보하기 위해, 프론트엔드 단에서 **바이트(Byte) 단위의 데이터 가공**을 수행했습니다. 오디오 파일의 헤더(Header)를 분석하여 포맷을 식별하고, iOS일 경우 비트 연산(Bitwise Operation)을 통해 좌/우 채널 데이터를 평균값으로 병합(Downmixing)하여 **Mono PCM16** 규격으로 변환해 전송하는 로직을 구현했습니다.
* **Conditional UX Design:** 앱 실행 시 `User Agent`를 분석하여 OS 환경을 감지하고, 이에 맞춰 UX 시나리오를 분기(Branching)했습니다. iOS 사용자에게는 '터치하여 소리 켜기' 오버레이 UI를 우선 노출하여, 사용자의 능동적인 액션을 유도함으로써 브라우저 정책을 준수하면서도 끊김 없는 미디어 경험을 제공했습니다.

**3) 결과**
* OS 및 브라우저 환경에 구애받지 않는 **100% 미디어 호환성** 확보.
* 클라이언트 리소스를 활용한 전처리로 서버 비용 절감 및 STT 인식 정확도 개선.

</div>
</details>
